## 2024-05-18 - Fix Insecure Direct Object Reference in Join Requests
**Vulnerability:** The API endpoint `/api/chats/[id]/requests` fetched a join request by `requestId` but failed to verify that the request actually belonged to the chat specified in the `[id]` route parameter. This allowed an admin of one chat to approve or reject a join request for an entirely different chat, bypassing authorization checks.
**Learning:** This is a classic Insecure Direct Object Reference (IDOR) pattern. While the code correctly verified the user was an admin of the *parent* context (the chat ID in the URL), it didn't verify the *child* resource (the join request) belonged to that parent context before mutating it.
**Prevention:** Always verify the relationship between child resources and their parent context when accessed via nested routing. Explicitly check `resource.parentId === route.parentId` before performing any operations on the resource.

## 2024-05-28 - Fix Insecure Direct Object Reference in Invite Code Retrieval
**Vulnerability:** The `GET` endpoint `/api/chats/[id]/invite` returned the invite code for any chat ID provided in the route parameters, without checking if the authenticated user was a participant of that chat. This allowed any authenticated user to retrieve the invite code for private chats or channels and subsequently join them, completely bypassing intended privacy controls.
**Learning:** Even simple read-only data endpoints that seem harmless (like fetching an invite code) can be critical security boundaries if the data grants access to other resources. Simply fetching data by a primary key (`chatId`) is insufficient if the data is subject to access control.
**Prevention:** Always verify the requesting user's authorization against the specific resource being accessed. When querying Prisma for a resource that requires membership access, explicitly join or filter on the participant relationship to ensure the user is authorized before returning sensitive properties.
## 2024-04-03 - [CRITICAL/HIGH] Fix Privilege Escalation via Mass Assignment in Role Update
**Vulnerability:** An Insecure Direct Object Reference (IDOR) and Mass Assignment vulnerability was discovered in the role update API (`PUT /api/chats/[id]/participants`). The endpoint accepted the `role` directly from the request body without validating it against an enum or allowed set of values (e.g., "ADMIN", "MEMBER"). Because `role` is defined as a `String` in Prisma rather than an ENUM, an "ADMIN" could escalate their own privileges or someone else's to "OWNER" by passing `{ "role": "OWNER" }`. Additionally, the endpoint did not enforce hierarchical restrictions, meaning any "ADMIN" could freely demote another "ADMIN", or worse, modify the role of an "OWNER".
**Learning:** When using Prisma `String` fields in place of native database ENUMs for access control properties (like roles or visibility), API endpoints must explicitly validate incoming values against an allowed set of constants. Furthermore, authorization logic must verify the target's current permissions before applying changes to enforce hierarchical rules (e.g., only the "OWNER" can modify another "ADMIN").
**Prevention:** Always validate user-provided data against a whitelist of acceptable values. Implement authorization checks that consider both the requester's and the target's roles before performing mutations.

## 2024-06-15 - [CRITICAL/HIGH] Fix Privilege Escalation in Participant Removal
**Vulnerability:** The `DELETE /api/chats/[id]/participants` endpoint failed to verify the role of the target participant being removed. It only checked if the requester was an `ADMIN` or `OWNER`. This allowed any `ADMIN` to remove other `ADMIN`s and even the `OWNER` of the chat, leading to an Authorization Bypass and Privilege Escalation.
**Learning:** Similar to mass assignment when updating roles, removing a resource with hierarchical permission levels requires checking the privileges of BOTH the requester AND the target.
**Prevention:** Always fetch the target entity in hierarchical permission structures before mutating or deleting it. Ensure business logic rules (e.g., "only an OWNER can remove an ADMIN", "an OWNER cannot be removed") are explicitly enforced.
## 2024-04-24 - Concurrent Authorization and Protected Query
**Vulnerability:** In `meowsenger/app/api/chats/[id]/requests/route.ts`, an authorization check (`prisma.chatParticipant.findUnique`) and a protected data query (`prisma.joinRequest.findUnique`) were executed concurrently using `Promise.all`.
**Learning:** This is a severe security anti-pattern. If the database executes the queries concurrently, the protected data is fetched *before* the authorization check resolves. This creates a race condition that could be exploited via timing attacks or subtle application logic errors to infer the existence or contents of protected records, even if the application eventually returns a 403 Forbidden.
**Prevention:** Authorization queries must strictly precede and fully resolve before any protected data queries are initiated. Use sequential `await` calls instead of `Promise.all` when authorization is a prerequisite for data access.
